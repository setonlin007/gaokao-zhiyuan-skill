# 研究纪要 · 真实数据格式实测（2026-06-05）

> 目的：在动手实现解析器/OCR 前，**实地核验**官方数据的真实发布格式与可靠性，避免"假设驱动"的设计。
> 方法：实抓官方/权威源 + 下载真实图片读图评估。所有结论附证据链接。
> 一句话结论：**格式因省、因数据集而异；按"数据可得性"选首发省（山东全 Excel）能基本绕开 OCR；OCR 是图片省（湖北）的核心而非全局核心。**

### 广东投档PDF 实测（curl 直连下载 + pypdf 抽取，铁证）

源：`https://eea.gd.gov.cn/attachment/0/585/585886/4746786.pdf`（67页，`/Image`=0、`DCTDecode`=0、`/Font`=140 → **纯文本，非扫描**）。抽取出的真实表头与数据：

```
院校代码 院校名称 专业组代码 计划数 投档人数 投档最低分 投档最低排位
10001 北京大学 206 34 35 689 99
10004 北京交通大学 208 90 90 620 13967
10004 北京交通大学(中外合作办学) 209 5 5 613 19398   ← 同校不同组差5000+位
```

- **一份 PDF 同时给到 Q1 两个核心字段：计划数 + 最低位次**，且**粒度=院校专业组**（院校代码-专业组代码），正是 DESIGN P0 缺陷2 要求的粒度。
- 解析只需一条正则（5位院校代码 + 院校名 + 5个整数列），**确定性、零 OCR**。
- 真实表头是 `投档最低排位`/`计划数`（非我 schema 假设的字段名）→ 列映射须按实际表头，再次印证"别假设"。
- 一分一段 PDF 实测列：`文化总分 | 本科[分数段人数, 累计人数] | 专科[分数段人数, 累计人数]`，Q1 取**本科累计人数**=位次；首行是"672（含以上）"聚合带 → 算术恒等校验须跳过首行（已如此实现）。

## 2. 抓取可靠性实测（都踩到了）

| 现象 | 证据 | 对设计的影响 |
|---|---|---|
| 官方 `hbea.edu.cn` 公告页**断连** | WebFetch socket closed ×2 | 需"已归档原件离线复算"兜底（已实现 `allow_local`）|
| 官方 `hbccks.cn` **TLS 证书无效 / http-only** | `ERR_TLS_CERT_ALTNAME_INVALID` | fetch **不强升 https + 容忍坏证书**（已修）|
| 阳光高考 `chsi.com.cn` **反爬** | 直取返回 **HTTP 412** | 中央平台不可裸抓，仅作发现/交叉核对 |

### ★ 根因：环境强制代理拦截 gov 站 TLS（已定位并解决）

`curl -v` 暴露真相：本环境带 `HTTPS_PROXY=http://129.211.81.198:31364`（腾讯云出口，即"VPN"）。代理对 gov 站的行为：

```
CONNECT eea.gd.gov.cn:443 → 200 隧道建立
TLS handshake Client hello → LibreSSL SSL_connect: SSL_ERROR_SYSCALL  ← 被重置
```

即**代理出口在 TLS 握手阶段被政务站重置**（商业 CDN 如 eol.cn 走代理却正常，是定向拦截）。

**解决：按用户指示"不走 vpn 而直连"——`curl --noproxy '*'` 绕过代理后**：

```
Trying 120.197.33.7:443 → Connected → TLSv1.2 握手成功
Server certificate: GlobalSign RSA OV SSL CA 2018 → verify ok
exit=0，文件下载成功
```

**结论**：
- 之前 WebFetch 的"断连/证书无效"**很大程度是这个代理造成的假象**；gov 站直连时 TLS 与证书其实正常。
- **生产抓取层必须支持 `no_proxy` 直连官方域**（或部署在境内）。`fetch.py` 已实现"对 allowlist 官方域强制绕过代理直连"。**（✅ 已落地）**

## 3. 阳光高考（chsi）作为中央平台的定位

- 是教育部指定**中央汇总平台**：有「各省一分一段汇总（陆续发布）」「省招办最新消息」「院校信息库」「招生计划」「志愿参考(阳光志愿)」。
  - 各省一分一段汇总：`gaokao.chsi.com.cn/gkxx/ss/202506/...`
  - 省招办消息：`gaokao.chsi.com.cn/gkxx/zc/ss`
- **但有反爬（412）**，且它多是**聚合各省原件链接/附件**，未必把数据归一化（湖北仍是图片、山东仍是 xls）。
- **用法定位**：优先用作 ①URL 发现（找到各省官方公告链接）②交叉核对（同一数字两个官方来源对账）；**不**作为可裸抓的统一数据 API（它不是 API）。

## 4. OCR 难度实测（读了真实湖北图片）

存证：`https://gaokao.eol.cn/hu_bei/dongtai/202506/W020250625632046485367.jpg`

- ✅ **有利**：清晰数字渲染（非扫描照片）、高对比、规整网格、印刷体数字 → 现代 OCR（PaddleOCR/表格模式）准确率应高。
- ⚠ **风险**：
  1. **斜向半透明水印**压在数字上（主要误读源）；
  2. **多列布局**：一页是 `分数|人数|累计` × 3 组 = 6 列数字，需正确切列与按分数降序重排；
  3. **6 张图拼接**成全省连续序列。
- ⚠ **真实表头是「人数」**（不是 schema 假设的「本段人数」）→ 列映射须按实际表头，再次印证"别假设"。
- ✅ **冗余可自检**：真实数据验证了恒等式 `累计[s] = 累计[s+1] + 人数[s]`（691→25；690 人数5→累计30；689 人数4→累计34…）。
  - **强结论**：一分一段表高度冗余——`累计`（=我们要的位次）可由 `分数 + 人数` 重算并与 OCR 出的 `累计` 列对账。即便水印污染部分 `累计` 单元格，也能**重算 + 交叉验证 + 定位错读**。OCR 在此场景**可做到可验证**（配双引擎 + 恒等式 + 人工只复核标红个位数）。

## 5. 对设计的修正建议（落地状态）

> 下列建议当时为"待确认"，现已大半落地——状态标在每条开头。

1. ✅**已落地**　**首发省选择标准改为"数据可得性优先"**：
   - 一级：关键数据是否 **文本PDF/Excel**（免 OCR）；二级：投档单位粒度是否正确。
   - → **首发选广东已落地**（一分一段+投档表全文本PDF、粒度=院校专业组本就对、官方站直连可达；投档表一份即给"计划数+最低位次"）。山东（Excel）次之，湖北（图片）留到 OCR 就绪。
   - DESIGN §4 已据此更新（判据=格式可得性，非改革与否），原"优先传统文理省"判据作废。

2. ✅**已落地**　**解析器优先级（按真实成本）**：`文本PDF（pypdf）` ≈ `Excel（xls→xlrd）` > `图片OCR`。
   - 广东全走文本PDF（`parse.py: parse_pdf_text` 已实现）；山东 `.xls`（待接 Excel 时再做）。
   - README §5 已改为"文本PDF/Excel 优先、OCR 仅图片省"。

3. ✅**已落地**　**一分一段位次重算策略**：以 `分数+人数` 重算 `累计` 并与原列对账；首行"（含以上）"聚合带跳过恒等校验（已如此实现）。

4. ✅**已落地**　**抓取层代理直连**：`fetch.py` 已对 allowlist 官方域强制绕代理直连（no_proxy，见 §2 根因）。

5. ⬜**待办（仅在接入《招生专业目录》时）**　**跟踪勘误**：广东《招生专业目录》有"更正及增补通知"（HTML），届时须把勘误并入，否则选科/计划有错。当前选科要求来自官方选考系统（另源），暂不受此影响。
